# 01 · 总体架构设计

## 1. 系统形态

模块化单体(Modular Monolith):一个代码库,模块间通过明确的接口交互,运行时按职责拆成少量独立进程。不引入消息队列、微服务、容器编排——个人系统的首要风险是"维护不动",不是"扛不住并发"。

## 2. 分层架构图

```
┌─────────────────────────────────────────────────────────────┐
│ 研究层  notebooks / vectorbt / qlib                          │
│ (自由探索,不受契约约束,产出:策略想法与参数)                  │
└───────────────────────────┬─────────────────────────────────┘
                            │ 想法成熟后,按契约改写
┌───────────────────────────▼─────────────────────────────────┐
│ 策略层  strategies/        ← 你的东西,插件目录               │
│ 每个策略 = 一个实现 StrategyBase 的类 + 一份实例 YAML 配置    │
└───────────────────────────┬─────────────────────────────────┘
                            │ 只依赖 03 号契约(事件 + Context)
┌───────────────────────────▼─────────────────────────────────┐
│ 核心层  core/                                                │
│  事件模型 / Context / StrategyBase / 引擎接口                 │
│  ┌─────────────────────┐   ┌──────────────────────────────┐ │
│  │ 回测引擎 backtest/   │   │ 实盘引擎 live/                │ │
│  │ 撮合·成本·绩效       │   │ OMS + 风控 risk/ + Gateway    │ │
│  └─────────────────────┘   └──────────────────────────────┘ │
└───────────────────────────┬─────────────────────────────────┘
┌───────────────────────────▼─────────────────────────────────┐
│ 数据层  data/                                                │
│ Provider(akshare/tushare/xtdata) → 清洗质检 → 存储           │
│ 存储:Parquet(行情) + DuckDB(查询) + SQLite(元数据/状态)      │
│ 对上提供统一 DataService:history()/get_bar()/calendar()      │
└─────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐

└─────────────────────────────────────────────────────────────┘
```

## 3. 两条工作流

**研究流(快)**:notebook 里随便折腾,vectorbt 向量化粗筛因子和参数,不讲究工程规范。
**生产流(严)**:想法定型后,按 03 号契约写成正式策略类 → 事件驱动回测验证(04)→ Paper 实盘(05)→ 真金实盘。只有走完生产流的策略才允许接真钱。

两条流共享数据层,这样研究和生产看到的是同一份数据,减少"研究时好好的、上线就不对"的来源。

## 4. 模块职责与自研/复用决策

| 模块 | 职责 | 决策 | 理由 |
|---|---|---|---|
| 数据采集 | 拉取行情/合约/复权因子 | **复用** akshare/tushare/xtdata,自己只写 Provider 适配器 | 爬数据是无底洞 |
| 数据存储 | 落盘、查询 | **复用** Parquet+DuckDB+SQLite,自己写薄封装 | 零运维,性能足够 |
| 事件模型+契约 | 策略与引擎之间的协议 | **自研**(几百行) | 这是系统的"接口主权",不能被某个框架绑死 |
| 回测引擎 | 事件回放、撮合、成本、绩效 | **自研**(中低频约 800~1500 行) | 自研才能保证撮合语义与实盘引擎一致;向量化粗筛另用 vectorbt |
| 实盘 OMS | 订单生命周期、持久化、对账 | **自研** | 核心资产,逻辑不复杂但必须完全可控 |
| 券商网关 | 对接 QMT/CTP/ccxt | **复用 SDK**(xtquant / vn.py 网关 / ccxt),自己写 Gateway 适配器 | 协议对接不值得自研 |
| 风控 | 前置检查、Kill Switch | **自研**(几百行) | 必须独立于策略且不可绕过 |
| 监控告警 | 日志、推送、日报 | **复用** loguru + webhook,自己写少量胶水 | — |

结论:你真正"实现系统"的工作量集中在**契约、回测引擎、OMS、风控**四块,其余都是适配器和胶水。

## 5. 运行时拓扑(三个进程,靠存储解耦)

| 进程 | 运行时机 | 做什么 | 失败影响 |
|---|---|---|---|
| 数据进程 `update_data.py` | 每个交易日收盘后(定时) | 增量更新行情→质检→写存储→发日报 | 不影响当天交易,影响次日 |
| 实盘进程 `run_live.py` | 交易时段 | 行情→策略→风控→网关,事件流水落盘 | 核心进程,见 07 故障恢复 |
| 研究环境 | 随时 | notebook / 回测脚本 | 无 |

进程之间**只通过文件存储和配置通信**,不搞进程间消息——简单、可重放、好排查。

## 6. 技术选型

| 类别 | 选择 | 备注 |
|---|---|---|
| 语言 | Python 3.11+ | 类型标注 + pydantic 校验 |
| 数据处理 | pandas / numpy / pyarrow | — |
| 行情存储 | Parquet 分区文件 + DuckDB 查询 | 列存,单机查询极快 |
| 元数据/状态 | SQLite | 合约表、日历、订单流水、KV 状态 |
| 配置 | YAML + pydantic | 启动时强校验,错配直接拒绝启动 |
| 日志 | loguru | 双轨:运行日志 + 事件流水(JSONL) |
| 调度 | OS 级定时(任务计划程序/cron)+ 进程内 APScheduler | — |
| 研究 | jupyter + vectorbt(可选 qlib) | 粗筛专用 |
| 实盘 SDK | xtquant(QMT)/ vn.py CTP / ccxt | 各包一个 Gateway |
| 测试 | pytest | 含黄金回归测试(见 04) |

## 7. 关键不变量(全系统必须始终成立)

1. 策略代码中**不允许出现** `if backtest / if live` 分支——行为差异只能由引擎实现承担。
2. 策略获取时间的唯一途径是 `ctx.now`,禁止 `datetime.now()`(回测/实盘不一致的头号来源)。
3. 所有订单必经风控模块,策略没有任何直达网关的路径。
4. 每笔订单、成交、风控拒绝、状态变更都持久化(SQLite + JSONL),系统的任何一天都可事后完整重放。
5. 实盘进程启动后的第一件事是**对账**(本地持仓 vs 券商持仓),对不上则冻结交易并告警,绝不带着错误状态下单。
