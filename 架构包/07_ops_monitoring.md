# 07 · 监控与运维

## 1. 日志双轨制

| 轨道 | 形式 | 内容 | 用途 |
|---|---|---|---|
| 运行日志 | loguru,按天滚动,保留 90 天 | 程序行为、调试信息 | 排查问题 |
| 事件流水 | JSONL,只追加,永久保留 | 每条订单/成交/拒单/风控动作/状态切换,含完整字段与时间戳 | 审计与重放:任何一天发生了什么,可逐事件还原 |

约定:策略内只用 `ctx.log()`,自动带上策略实例 id;严禁 print。

## 2. 告警矩阵

| 级别 | 渠道(示例) | 典型事件 |
|---|---|---|
| INFO | 每日汇总推送 | 数据更新完成、日报 |
| WARN | 企业微信/钉钉 webhook 即时 | 质检异常、单笔拒单、行情短暂中断 |
| CRIT | webhook + 短信/电话(可用第三方转发) | 对账失败、Kill Switch 触发、实盘进程崩溃、网关断连超时 |

告警去重:同类告警 N 分钟窗口内合并,避免风暴把真正重要的淹没。**每月做一次告警演练**(人为触发一条 CRIT,确认手机真的收到了)。

## 3. 每日任务时刻表(A股示例)

| 时间 | 任务 | 失败处置 |
|---|---|---|
| 08:45 | 盘前自检:数据可用、配置校验、网关连通、启动对账 | 任一失败 → CRIT,当日不自动开启交易 |
| 09:15 | 实盘进程进入交易时段 | — |
| 15:05 | 收盘:撤残单、停止策略、收盘对账 | 对账失败 → CRIT |
| 16:30 | 数据进程:增量更新 + 质检 | 失败 → WARN,重试;再失败 → CRIT |
| 17:30 | 日报推送:当日 PnL、持仓、成交、拒单、数据质检摘要 | — |
| 23:00 | 备份任务 | 失败 → WARN |

调度用 OS 级(Windows 任务计划程序 / cron)拉起脚本,进程内细粒度定时用 APScheduler。

## 4. 部署

- **最小拓扑**:一台 Windows 实盘机(QMT 要求)跑实盘进程 + 数据进程;研究在你日常电脑上,通过同步盘/Git 共享代码与数据目录。
- 实盘机要求:有线网络、UPS 或选用云 Windows 主机、关闭自动更新重启、NTP 时间同步(时间漂移会造成委托时段误判)。
- 进程守护:Windows 用 NSSM 把实盘进程注册为服务(崩溃自动拉起,拉起后照例先对账);Linux 侧(期货/crypto)用 systemd。
- 升级流程:改动 → 测试全绿(含黄金回归)→ 收盘后部署 → 次日盘前自检通过才生效。**严禁盘中热更新代码**。

## 5. 备份与密钥

- 每日备份:meta.db、事件流水、配置目录 → 压缩后推到网盘/对象存储;Parquet 行情可重建,周备即可。
- 每季度做一次**恢复演练**:在干净环境用备份把系统拉起来——没演练过的备份等于没有。
- 密钥(账号、token):放 `.env` 或系统凭据管理器,严禁进 Git;`.gitignore` 第一天就配好。

## 6. 故障恢复 Runbook(打印贴在桌上)

1. 实盘进程崩溃/失联 → 确认进程状态;查看最近 CRIT 告警与运行日志尾部。
2. 登录券商官方客户端/APP,**人工确认当前真实持仓与挂单**(任何时候,券商端是事实标准)。
3. 有危险敞口且系统短时间起不来 → 直接在官方客户端手工处理,不要硬等系统。
4. 重启实盘进程 → 自动进入启动对账 → 对账通过自动恢复;不通过保持 FREEZE,人工核对差异来源(常见:崩溃瞬间的在途成交回报丢失,以券商记录修正本地)。
5. 事后:从事件流水定位崩溃前最后事件,写 3 行事故记录(现象/原因/改进),改进项进任务列表。
