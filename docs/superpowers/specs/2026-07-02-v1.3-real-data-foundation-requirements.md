# v1.3 Real Data Foundation Requirements

Status: `REQUIREMENTS_DRAFT`
Owner: Quant CIO
Date: 2026-07-02
Scope: personal quant research platform data foundation

This document is a CIO-level requirements package for the next data-platform
iteration. It is not an implementation plan, investment advice, trading
permission, M4/QMT approval, broker integration approval, or real-money
authorization.

The implementation focus is Phase 1, split into Phase 1A and Phase 1B to avoid
turning the first implementation pass into a provider grab bag. Phase 2 and
Phase 3 define the upgrade roadmap so Phase 1 can be built without painting the
platform into a corner.

## 1. CIO Decision

Decision: `START_PHASE_1A_DESIGN_WITH_WARNINGS`

Recommended next action: ask Superpowers to write the Phase 1A design and
implementation plan first: data lake, manifest/calendar contract, provider
probe, gitignore/data isolation, and an A-share/ETF minimum closed loop. Phase
1B should add the crypto provider closed loop only after the Phase 1A contracts
are stable.

Default safe option: all new datasets remain `research-only` until they pass
data audit. No strategy may use new data for paper/live promotion without the
full strategy promotion workflow: thesis, data audit, backtest validation, risk
review, paper observation, paper/live gate, and human decision.

Not allowed:

- Do not connect broker gateways, QMT, exchange trading APIs, or real accounts.
- Do not store real credentials, tokens, account identifiers, or live overlays
  in git.
- Do not describe any dataset as live-money ready before sustained data quality
  evidence exists and the repo gates are satisfied.
- Do not use backtest profitability to weaken data, risk, or paper/live gates.

## 2. Problem Statement

The current repository has useful early real-data work for A-share ETFs and
research packages for ETF, A-share individual stocks, and crypto. The data layer
is still strategy-adjacent: each research line can require bespoke data
collection, manual inspection, and local evidence.

The platform needs a general data foundation that can support recurring CIO
research loops:

```text
ingest real data
-> normalize into common contracts
-> audit quality
-> publish research-ready snapshots
-> feed backtests and strategy research
-> block unsafe or stale data automatically
```

The goal is not to chase the largest possible data vendor list. The goal is a
maintainable, auditable, reproducible, low-cost data base that can be upgraded
vendor by vendor without changing strategy code or weakening gates.

## 3. Target Outcomes

Phase 1 must deliver these outcomes in two ordered subphases:

Phase 1A: data lake and A-share/ETF minimum closed loop.

- A consistent local data lake layout for raw, staged, curated, and audit
  artifacts.
- A normalized dataset contract usable by the existing research/backtest stack.
- A manifest lifecycle schema that can represent fetched, staged,
  curated-candidate, blocked, and research-ready states.
- A provider probe and field-availability matrix for Tushare Pro, AKShare, and
  BaoStock.
- A-share/ETF daily data ingestion for the minimum universe defined below.
- AKShare and/or BaoStock cross-checks for close and volume on a representative
  sample.
- A daily update mechanism for Phase 1A with deterministic logs, hashes,
  manifests, and quality reports.
- A data-audit gate that classifies each Phase 1A dataset as `PASS`,
  `PASS_WITH_WARNINGS`, or `FAIL`.

Phase 1B: crypto provider closed loop.

- Provider adapters for Binance and OKX public market data.
- Coinbase and Kraken sanity checks where accessible and symbol-compatible.
- 4h and 1d crypto datasets for the minimum universe defined below.
- Crypto-specific timestamp, venue, and cross-source quality checks.
- A daily update mechanism with deterministic logs, hashes, manifests, and
  quality reports.
- A clear distinction between `fetched`, `staged`, `curated_candidate`,
  `research_ready`, and any future `paper_candidate` state.
- Documentation sufficient for Superpowers to create an implementation design
  and plan without re-litigating the CIO scope.

## 4. Phase 1 Decomposition and Required Order

Phase 1 is intentionally split.

### 4.1 Phase 1A: Data Lake + Manifest + A-Share/ETF Minimum Closed Loop

Phase 1A is the first implementation target.

Required order:

1. Add data-lake and local-secret path isolation requirements to `.gitignore`
   through the implementation plan.
2. Extend or wrap the existing manifest/calendar contract so A-share exchange
   calendars are supported before provider adapters depend on them.
3. Implement provider probes and field-availability matrices for Tushare Pro,
   AKShare, and BaoStock.
4. Build the minimum A-share/ETF daily ingestion, staging, quality, and audit
   flow.
5. Publish only `curated_candidate` snapshots until data audit marks them
   `research_ready`.

Phase 1A success is a working A-share/ETF daily pipeline for a small,
auditable universe, not full market coverage at any cost.

### 4.2 Phase 1B: Crypto Provider Closed Loop

Phase 1B starts after Phase 1A contracts and quality reports are stable.

Required order:

1. Reuse the Phase 1A data lake, manifest lifecycle, hash, and quality-report
   contracts.
2. Add Binance and OKX provider probes.
3. Add Coinbase and Kraken sanity probes where available.
4. Build 4h and 1d crypto ingestion for the minimum universe.
5. Add venue/timestamp/cross-source quality checks.

Phase 1B must not fork the data contract into a crypto-specific side system.

## 5. Non-Goals

Phase 1 does not include:

- Real-money trading.
- QMT, broker, or exchange order gateway integration.
- Live order routing or account reconciliation against a real account.
- Full-market A-share provider coverage before Phase 1A proves the minimum
  closed loop.
- Minute, tick, Level-2, order-book history for A-shares.
- Paid institutional feeds such as Wind, Choice, iFinD, CSMAR, RESSET, Kaiko,
  Coin Metrics, or CoinDesk Data.
- A web UI.
- A large external data warehouse or cloud object-store dependency.
- A new standalone repository unless Superpowers determines the current repo
  cannot safely contain the module.

Recommended placement: implement Phase 1 as an internal data-foundation
subsystem inside the current `quant` repository.

Suggested target shape:

```text

  contracts, manifests, quality checks, readers

src/quant/data_providers/
  tushare, akshare, baostock, binance, okx, coinbase, kraken adapters

  ingest, rebuild, audit, daily_update, provider_probe commands

  provider configs, universe configs, schedule configs, quality thresholds

docs/runbooks/
  data daily runbook, incident response, source onboarding checklist

research/data_audits/
  human-readable data audit reports

data_lake/
  local raw/staged/curated/audit outputs; gitignored
```

Before implementation writes any real provider output, `.gitignore` or an
equivalent repository guard must exclude:

```text
data_lake/
config/data/local/
.env
.env.local
.env.*.local
!.env.example
```

The current repository already ignores `runtime/`, `results/`, `data_real/`,
and `config/live/local/`; Phase 1A must explicitly add the data-lake and local
provider-secret paths through the implementation plan.

Implementation may choose slightly different names, but it must preserve the
same ownership boundaries:

- strategy code consumes curated data through contracts;
- provider code does not leak into strategies;
- raw data and credentials stay out of git;
- audit evidence remains reproducible and discoverable.

## 7. Data Lifecycle States

Every dataset must move through explicit states:

| State | Meaning | Allowed use |
| --- | --- | --- |
| `fetched` | Raw provider output exists | Inspection only |
| `staged` | Raw output normalized to common schema | Structural tests only |
| `curated_candidate` | Dataset has manifest, provider metadata, hashes, and quality report, but has not passed data audit | No backtest claim; audit review only |
| `research_ready` | Data audit returned `PASS` or accepted `PASS_WITH_WARNINGS` | Backtest/research allowed |
| `paper_candidate` | Strategy-specific data, backtest, and risk evidence are complete | Paper review only |
| `blocked` | Missing, stale, inconsistent, or unaudited | No strategy use |

Phase 1 must support these states in manifests and reports. It does not need to
support paper/live promotion.

All new provider data defaults to `fetched`, `staged`, or
`curated_candidate`. No dataset may be called `research_ready` until the data
audit gate explicitly records `PASS` or accepted `PASS_WITH_WARNINGS`.

## 8. Canonical Storage Layers

The data lake must have four layers.

### 8.1 Raw

Purpose: preserve provider output exactly enough to reproduce and inspect
normalization.

Required contents:

- Original response payloads or downloaded files where practical.
- Provider name, endpoint, request parameters, response status, ingest time.
- Source license note and token-free provider identifier.
- File hashes and row counts.

Rules:

- Raw files are append-only.
- Raw files must not be overwritten silently.
- Raw files must not include API keys, auth tokens, or account identifiers.

### 8.2 Staged

Purpose: convert provider-specific fields to common typed columns.

Required properties:

- Stable schema per dataset type.
- Explicit timezone and timestamp semantics.
- No strategy-specific features.
- No forward-filled values unless the fill policy is documented in the quality
  report.

### 8.3 Curated

Purpose: publish research-consumable snapshots with clear universe, calendar,
adjustment, and quality semantics.

Required contents:

- `dataset_manifest.yaml` or equivalent structured manifest.
- Parquet data files or equivalent columnar storage.
- Provider metadata.
- Quality report.
- Hash report.
- Universe definition.
- Calendar and adjustment policy.

### 8.4 Audit

Purpose: explain whether a dataset can be trusted for the intended use.

Required outputs:

- Machine-readable quality result.
- Human-readable markdown report.
- Blocking issues and warnings.
- Checks performed.
- Required fixes.

## 9. Technology Requirements

Phase 1 should use:

- Parquet for large tabular data.
- DuckDB for local analytical queries and validation.
- CSV only for small fixtures, compatibility exports, and human-readable
  samples.
- YAML or JSON for manifests and quality metadata.
- Existing Python project tooling unless Superpowers finds a strong reason to
  add a dependency.

Phase 1 should avoid:

- A mandatory external database server.
- A mandatory cloud account.
- Notebook-only workflows.
- Provider-specific schemas exposed to strategy code.

## 10. Phase 1 Data Scope

### 10.1 A-Share and ETF Scope

Primary provider: Tushare Pro configured permission tier, discovered by
`provider_probe` before ingestion.

Cross-check providers: AKShare, BaoStock.

Tushare permissions must not be assumed from the phrase "basic permissions."
Phase 1A must generate a field-availability matrix before declaring scope:

| Field group | Example datasets | Required Phase 1A behavior |
| --- | --- | --- |
| instruments and trade calendar | stock/fund basic metadata, calendar | probe availability and record permission result |
| daily bars | raw daily OHLCV for A-share/ETF | probe availability and record provider function/endpoint |
| adjustment factors | ETF/fund adjustment such as `fund_adj` where used | probe permission; if unavailable, block adjusted-price research |
| price limits | A-share daily limit-up/down such as `stk_limit` where used | probe permission; if unavailable, mark limit-up research data blocked |
| daily basic/fundamental fields | turnover, market cap, PE/PB or provider equivalents | probe permission; mark unavailable fields explicitly |

The field matrix must record: provider, API/function, requested field group,
permission result, observed columns, sample date range, row count, and failure
reason if unavailable.

Minimum Phase 1A universe:

- Metadata-only universe discovery:
  - Full active A-share instrument list if available from primary provider.
  - Full ETF/fund instrument list if available from primary provider.
  - These lists are for instrument metadata and survivorship checks only in
    Phase 1A; they do not imply full-market daily bars in the first pass.
- First-pass daily bars universe:
  - `510300.SH`
  - `510500.SH`
  - a small configurable A-share sample, with the implementation plan choosing
    a bounded size such as 10-20 symbols.

Phase 1A must not attempt full-market A-share or ETF daily bars until the
minimum closed loop passes data audit on the first-pass universe.

Required A-share/ETF datasets:

| Dataset | Required fields | Phase 1 use |
| --- | --- | --- |
| `instruments` | symbol, exchange, name, asset type, listing date, delisting date when available, status | universe and survivorship checks |
| `trade_calendar` | date, exchange, open/closed status | daily session alignment |
| `bars_1d_raw` | symbol, session date/time, open, high, low, close, volume, amount | raw-price event and execution assumptions |
| `adjust_factors` | symbol, date, factor, factor type/source | adjusted-price research |
| `bars_1d_adjusted` | symbol, date/time, adjusted open/high/low/close when available or reproducible | adjusted trend/factor research |
| `suspensions` | symbol, date, suspension/resumption state when available | tradability checks |
| `st_status` | symbol, date or effective interval, ST/special treatment flag when available | universe filters |
| `price_limits` | symbol, date, limit-up, limit-down when available or computed with documented rule | limit-up/down research |
| `daily_basic` | turnover, float market cap, total market cap, PE/PB if available | factor and liquidity screens |

If a field is unavailable from the chosen provider tier, Phase 1 must mark the
field as unavailable in the manifest rather than silently omitting the risk.

### 10.2 Crypto Scope

Primary source candidates:

- Binance public market data.
- OKX public market data.

Sanity sources:

- Coinbase public/advanced market data where accessible.
- Kraken public market data where accessible.

Minimum Phase 1 universe:

- `BTCUSDT`
- `ETHUSDT`
- `SOLUSDT`

Required crypto datasets:

| Dataset | Required fields | Phase 1 use |
| --- | --- | --- |
| `instruments` | symbol, base asset, quote asset, venue, status, tick size, lot size, min notional when available | tradability and sizing assumptions |
| `bars_1d` | open time, close time, OHLC, volume, quote volume, trade count, source | daily trend and sanity checks |
| `bars_4h` | open time, close time, OHLC, volume, quote volume, trade count, source | 4h strategy research |
| `cross_source_prices` | symbol, timestamp, source, close, comparison result | venue sanity checks |
| `symbol_status` | symbol, effective time, trading status when available | listing/delisting awareness |

Crypto timestamp policy must be explicit:

- Exchange open time is the provider identity key.
- Engine-facing decision time must be the fully closed bar boundary.
- The manifest must state whether `dt` stores open time, close time, or a
  normalized decision timestamp.

## 11. Provider Requirements

Each provider adapter must expose these capabilities where applicable:

```text
probe_connection()
probe_permissions()
probe_field_availability()
list_instruments()
fetch_history(dataset, symbols, start, end, frequency)
fetch_incremental(dataset, symbols, since)
normalize(raw_path)
write_provider_metadata()
```

Provider adapters must not:

- make strategy decisions;
- write directly into curated snapshots without staged validation;
- log secrets;
- hide partial failures.

Provider output must include:

- provider name;
- endpoint or API function;
- request parameters;
- ingest timestamp;
- provider version or client package version when available;
- source timezone;
- row count;
- content hash.

Provider probe output is a Phase 1A deliverable. It must be available before a
dataset requirement is treated as implementable. If a provider account cannot
access a required field group, the resulting dataset must be marked
`blocked` or `curated_candidate` with `missing_fields` and `known_limitations`
recorded in the manifest, not `research_ready`.

## 12. Daily Update Requirements

Phase 1 must include a daily update command or commands that can be scheduled by
cron, launchd, or a future automation layer.

Recommended update flow:

```text
load config
-> verify provider credentials / connectivity
-> fetch raw increment
-> normalize staged data
-> merge into curated snapshot candidate
-> run quality checks
-> write manifest/hash/quality report
-> publish snapshot if gates pass
-> mark blocked if gates fail
```

Daily update windows:

- A-share/ETF daily data: run after mainland China close and provider update
  window. The initial recommended window is 17:30-20:30 Asia/Shanghai.
- Crypto 4h data: can run after each 4h UTC close plus a confirmation delay.
- Crypto daily data: can run after UTC daily close plus a confirmation delay.

The exact schedule should be configurable. No strategy should assume fresh data
until the quality report says the expected latest session is present.

Phase 1A only needs the A-share/ETF daily update path. Phase 1B adds crypto 4h
and 1d update paths after the shared lifecycle and manifest contracts are in
place.

## 13. Quality Requirements

All `curated_candidate` and `research_ready` datasets must run deterministic
checks.

Common checks:

- provider is named;
- build command is known;
- universe is explicit;
- date range is explicit;
- timezone is explicit;
- no duplicate `(symbol, dt, frequency)` rows;
- required OHLCV columns exist;
- OHLC is non-negative and internally plausible;
- volume and amount are non-negative;
- no rows beyond the declared decision timestamp;
- latest expected session is present or explicitly marked missing;
- all manual repairs are recorded.

A-share/ETF checks:

- bars align with A-share trade calendar;
- non-trading days are absent unless explicitly marked;
- adjustment factors exist for adjusted-price datasets;
- instrument exchange suffixes are stable;
- ST, suspension, and delisting fields are available or marked unavailable;
- price-limit data is available, computed with documented rules, or marked
  unavailable;
- raw and adjusted price semantics are not mixed silently.

Crypto checks:

- 4h and 1d interval boundaries are complete;
- exchange open-time inclusivity is handled deterministically;
- cross-source close differences are reported;
- quote currency is explicit;
- symbol trading status is recorded where available;
- 7x24 calendar does not reuse A-share holiday logic.

## 14. Data Audit Gate

Each dataset must produce a data-audit verdict:

```text
PASS
PASS_WITH_WARNINGS
FAIL
```

Hard-fail conditions:

- unknown data source;
- missing trade calendar for A-share/ETF daily datasets;
- missing or unexplained adjustment factors for adjusted-price research;
- duplicate primary keys in curated bars;
- future rows relative to declared simulation/decision time;
- quality issues without a report;
- stale latest session without an explicit blocked state;
- manual repair without an audit note.

`PASS_WITH_WARNINGS` is acceptable for research when warnings are documented and
do not invalidate the intended use. It is not automatic paper/live approval.

## 15. Dataset Manifest Requirements

Every `curated_candidate` or `research_ready` snapshot must include a manifest
with at least:

```yaml
schema_version: string
dataset_id: string
asset_class: a_share | etf | crypto
lifecycle_state: fetched | staged | curated_candidate | research_ready | paper_candidate | blocked
universe_id: string
frequency: 1d | 4h | other
as_of: timestamp
decision_time:
  timestamp: timestamp
  policy: fully_closed_bar | exchange_session_close | provider_publish_time | manual_snapshot
coverage:
  start: timestamp_or_date
  end: timestamp_or_date
timezone: string
calendar: string
adjustment_policy: raw | qfq | hfq | none | mixed_with_fields
build:
  command: string
  code_version: string
  generated_at: timestamp
source:
  provider_primary: string
  provider_cross_checks: []
  endpoint_or_function: string
  request_params: {}
  provider_version: string_or_null
  retrieved_at: timestamp
license:
  usage_scope: research_only | unknown | contracted
  source_license_note: string
  redistribution_allowed: false
storage:
  raw_path: path
  staged_path: path
  curated_path: path
quality:
  verdict: PASS | PASS_WITH_WARNINGS | FAIL
  report_path: path
  generated_at: timestamp
  lifecycle_state_after_audit: fetched | staged | curated_candidate | research_ready | blocked
hashes:
  manifest_sha256: string
  data_sha256: string_or_map
known_limitations: []
missing_fields: []
```

Implementation may extend this schema, but must not remove these concepts.

The manifest must carry enough information to reproduce why a dataset was
published or blocked. Source and license details must be top-level sections,
not hidden under hashes.

## 16. Security and Secrets

Phase 1 must support provider tokens without committing them.

Requirements:

- Use environment variables or local ignored config for tokens.
- Provide example config with placeholder names only. Example files may use
  `.env.example` or `*.example.yaml` and must remain safe to commit.
- Do not write tokens to logs, manifests, raw payloads, reports, or test
  fixtures.
- Add `data_lake/`, `config/data/local/`, `.env`, `.env.local`, and
  `.env.*.local` to repository ignore rules before writing real provider data
  or credentials. If a broad `.env.*` ignore rule is used, it must explicitly
  unignore `.env.example`.

## 17. Error Handling and Incident States

Daily update must classify failures.

Required categories:

| Category | Meaning | Default action |
| --- | --- | --- |
| `provider_unavailable` | source API failed or timed out | keep previous curated snapshot; mark update failed |
| `partial_fetch` | some symbols/frequencies missing | block new snapshot unless configured as non-critical |
| `schema_changed` | provider columns changed unexpectedly | block and require review |
| `quality_failed` | deterministic checks failed | block and write report |
| `cross_source_divergence` | sources disagree beyond threshold | block or warn based on severity |
| `stale_data` | expected latest session missing | block strategy use for affected dataset |
| `manual_repair_required` | automated repair is unsafe | block until documented review |

No failure should silently publish a new `research_ready` snapshot.

## 18. Integration Requirements

Phase 1 must integrate with existing repository boundaries:

- The existing manifest/calendar contract must be extended or wrapped before
  A-share provider ingestion depends on it. The current manifest reader supports
  `continuous_24x7` calendar semantics; A-share exchange sessions need explicit

- Existing backtest and research code should consume curated datasets through
  `quant.data` contracts.
- Strategies must not import provider adapters.
- The backtest layer should be able to identify the dataset manifest used for a
  run.
- Research reports should be able to reference dataset IDs and quality verdicts.
- CIO research packages should cite data audit artifacts before backtest claims.

## 20. Phase 2 Roadmap: Minutes and Higher Frequency

Phase 2 should start only after Phase 1 daily/4h data is stable.

Candidate additions:

- A-share historical minute data from Tushare.
- Trial comparison of JQData and RQData for A-share minute/tick coverage,
  field quality, cost, licensing, and API stability.
- Crypto minute data from existing exchange APIs.
- More explicit execution-cost and slippage research datasets.

Phase 2 must answer:

- Is minute data complete enough for the intended research?
- Are timestamps session-aligned and timezone-safe?
- Does the provider include suspensions, partial sessions, and corporate-action
  adjusted minute semantics?
- Can the data volume remain manageable locally?
- What additional quality checks are needed for missing intraday intervals?

Phase 2 does not automatically unlock same-day board-capture, high-frequency, or
live execution research. Those require separate strategy, execution, and risk
requirements.

## 21. Phase 3 Roadmap: Institutional and Research-Grade Feeds

Phase 3 should be treated as a procurement and governance upgrade, not a simple
adapter task.

Candidate upgrades:

- CSMAR or RESSET for academic-grade fundamentals, delisted securities, and high
  frequency history.
- Wind, Choice, or iFinD for professional terminal/API coverage.
- Kaiko, Coin Metrics, or CoinDesk Data for institutional crypto history,
  exchange normalization, and audit-grade coverage.

Phase 3 entry criteria:

- Phase 1 data foundation is stable.
- There is a documented research need that free/low-cost sources cannot satisfy.
- Cost, licensing, export rights, redistribution limits, and credential handling
  are reviewed.
- A vendor comparison report exists.

Phase 3 acceptance must include a contract/license summary in the dataset
manifest and a migration plan that preserves existing dataset IDs or maps them
explicitly.

## 22. Open Design Decisions for Superpowers

Superpowers should decide during design:

1. Exact module and command names.

## 25. Final CIO Summary

The recommended next iteration is `v1.3-real-data-foundation`, starting with
Phase 1A design.

It should be implemented as a core platform capability in this repository, with
local data lake outputs kept out of git. Phase 1A should first produce the data
lake, manifest lifecycle, provider probe, and A-share/ETF minimum closed loop.
Phase 1B should then add the crypto provider closed loop using the same
contracts. Phase 2 and Phase 3 are intentionally planned but not implemented
now.

The success condition is not "we downloaded data." The success condition is:

```text
Every research dataset has a source, manifest, hash, quality report, audit
verdict, update path, and failure mode.
```

Until that condition is met, datasets remain fetched artifacts, not trusted
research inputs.
