先记住三句话

第一，回测 是用历史数据检查策略想法。

第二，Paper 是模拟盘，不花真钱，用来演练真实流程。

第三，M3b 是真钱前观察期，没完成前不要接实盘。

一、这个系统现在到底是什么状态

你可以把项目理解成一座正在分阶段建造的交易系统。现在已经可以在本地训练和验证，但还没有开放真实券商接口。

M0

项目骨架

确定目录、配置、基础模块和测试方式，让系统能被持续开发。

M1

数据层

能读取样例行情、交易日历、标的信息，并做基础质量检查。

M2

可信回测

能把策略放到历史日线数据里跑，生成订单、成交和权益曲线。

M3 当前阶段

Paper 模拟盘

不碰真钱，演练订单、风控、对账、告警和故障恢复。

M4 暂时禁止

QMT / 实盘

M3b 没有连续观察签收前，不接真实券商，不实盘下单。

风险边界：现在不要把它当实盘机器人

当前项目明确排除了 QMT、真实券商网关、真钱交易、分钟线和 Web UI。你现在能做的是学习、回测、模拟、观察和演练。

二、先把金融和系统术语翻译成人话

术语听起来吓人，其实大多数都能翻译成生活语言。先理解这些词，后面的命令就不会像咒语。

回测

用过去的行情数据，假装自己回到当时，看策略会不会买、会不会卖、最后资金变成多少。

Paper 是什么

Paper 不是纸，是“模拟盘”。它像交易演练场：流程像真的，但不花真钱，也不连接真实券商。

策略

一套买卖规则。比如：短期均线比长期均线高就买，低就卖。

标的

你要交易的东西。这里的样例标的是 510300.SH，可以理解成一只 ETF。

K 线 / 日线

把一天的开盘价、最高价、最低价、收盘价整理成一条记录。日线就是一天一条。

均线

最近几天收盘价的平均值。短均线看近期变化，长均线看稍长趋势。

订单

系统发出的“我要买”或“我要卖”的请求。订单不一定马上成交。

成交

订单真的买到或卖出了，才叫成交。回测里可能有订单但没有成交。

仓位

你现在手里持有多少标的。持仓 0 就是空仓，持有 10000 份就是有仓位。

风控

防止系统乱买、买太多、状态异常还继续买。风控是刹车，不是赚钱按钮。

对账

检查系统记录和账户记录是否一致。现金差异、持仓差异都要看。

CRIT 告警

CRIT 是严重告警。比如网关断连，系统要立刻提醒并冻结新开仓。

NORMAL

系统状态正常。Paper 跑完后看到这个状态，表示没有停在危险状态。

FREEZE_OPEN

冻结开仓。系统可以保护自己，不再接受新的买入开仓动作。

HALT

更严肃的暂停状态。遇到未知风险时先停下来，查清楚再恢复。

M3b

真钱前 Paper 观察期。至少 10 个交易日，每天对账零差异，还要做断连和告警演练。

M4

未来接 QMT / 真实券商的阶段。M3b 没签收前，M4 是被阻塞的。

runtime

系统运行后产生的记录目录，里面有事件日志、状态库、告警投递文件等。

三、项目地图：每个目录是干什么的

小白最容易迷路。先看地图：你日常主要碰 config/、strategies/、scripts/ 和 runtime/。

  • 个人量化系统/ 工作区根目录
  • quant/ 项目主体
    • README.md 当前阶段说明
    • config/ 配置：资金、策略、风控、成本
    • strategies/ 策略代码，例如双均线
    • scripts/ 你会运行的命令入口
    • data_sample/ 样例行情数据
    • runtime/ 运行后生成的记录
    • docs/runbooks/ Paper 每日操作手册
    • tests/ 自动测试，确认系统没坏
  • 教程/ 本教程所在目录

你最常用的 3 个脚本

脚本 作用
scripts/run_backtest.py 跑回测，用历史数据检查策略。
scripts/run_paper.py 跑 Paper 模拟盘，演练接近真实交易的流程。
scripts/ops.py 查状态、冻结开仓、暂停、恢复，是运维刹车工具。

不要随便删 runtime

如果只是自己练习，可以清理旧运行记录；如果正在做连续 M3b 观察，就不要随便删除，否则观察链条会断。

四、第一次使用：先进入正确目录

所有命令都从项目主体目录运行。你现在这台机器上已经有 .venv 虚拟环境，所以最稳的命令是 .venv/bin/python。

为什么不用 python？

有些 macOS 环境没有 python 命令，直接敲会出现 command not found: python。 这个项目的虚拟环境在 quant/.venv/，用 .venv/bin/python 可以直接指定正确的 Python。

进入项目主体目录 
cd /Users/shenxi/Desktop/WORK-SPACE/个人量化系统/quant

这条命令的意思：先走到真正放代码的 quant 目录里。

检查 Python 版本 
.venv/bin/python --version

你应该看到类似 Python 3.13.3 的输出。

检查系统是否健康 
.venv/bin/python -m pytest -q

我实测时看到 120 passed。这表示当前代码和测试是健康的。

回测小报告

最简单的结果摘要。当前样例数据很短，重点是验证流程，不是看收益。

orders.csv

订单表

策略发出了哪些买卖请求。订单可能提交了，但不一定成交。

trades.csv

成交表

真正买到或卖出的记录。空文件表示这次没有实际成交。

equity.csv

权益曲线

每天总资产和现金是多少。以后样例数据更长时，这里会更有意义。

events.jsonl

事件流水

机器可读的事件记录，一行一个 JSON，适合追踪系统到底做过什么。

config_snapshot.yaml

配置快照

保存当时用的策略配置。以后复盘时，知道这次结果是用什么参数跑出来的。

如何解读“有订单但没成交”

订单是“我想买”，成交是“我真的买到了”。样例数据只有几行，出现订单但没有成交是正常的。 这不代表系统坏了，只代表这次回测的样例条件下没有撮合出成交。

六、策略：双均线策略到底在想什么

当前示例策略叫 dual_ma，意思是 dual moving average，双均线。它不是神奇模型，只是一套非常经典的趋势规则。

策略规则翻译

代码里的话 人话
fast: 3 看最近 3 天的平均收盘价，代表短期走势。
slow: 5 看最近 5 天的平均收盘价，代表稍长一点的走势。
target_qty: 10000 如果要买，就希望持有 10000 份。
ma_fast > ma_slow 短期平均价高于长期平均价，理解成短期趋势更强，于是买入。
ma_fast < ma_slow 短期平均价低于长期平均价，理解成趋势走弱，于是清仓。

策略简单不代表一定赚钱

双均线只是教学用的第一套规则。真正策略要经过更长数据、更严格成本、滑点、风控、回撤和样本外验证。 当前阶段的重点是学会系统流程。

七、Paper：真钱前的模拟盘演练

Paper 模式可以理解成“穿着救生衣练游泳”。它不接真实券商，不花真钱，但会认真记录订单、状态、对账和告警。

Paper 不是真钱

Paper 的价值不是赚钱，而是检查交易系统有没有纪律：启动前是否对账、异常时是否冻结、告警是否可见、恢复是否有依据。

运行 Paper 模拟盘 
cd /Users/shenxi/Desktop/WORK-SPACE/个人量化系统/quant
.venv/bin/python scripts/run_paper.py \
  --strategy config/strategies/dual_ma_510300_paper.yaml \
  --paper config/paper.yaml \
  --max-bars 20

你应该看到类似 paper events: runtime/paper/events.jsonl、paper store: runtime/paper/meta.db、final state: NORMAL。

查看 Paper 当前状态 
cd /Users/shenxi/Desktop/WORK-SPACE/个人量化系统/quant
.venv/bin/python scripts/ops.py \
  --store runtime/paper/meta.db \
  --events runtime/paper/events.jsonl \
  --operator shenxi \
  status

正常情况下输出 NORMAL。如果不是 NORMAL，不要急着继续跑，要先看运维章节。

runtime/paper/events.jsonl 是流水账

每发生一件重要事情，就写一行事件。比如启动对账、断连、告警、恢复。它适合追溯“系统到底做过什么”。

runtime/paper/meta.db 是状态库

SQLite 数据库，保存 Paper 当前状态、订单和成交等结构化记录。它适合查询“系统现在是什么状态”。

八、运维：学会刹车，比学会开车更重要

真正交易系统最怕“状态不清楚还继续跑”。所以 M3 阶段已经有状态机和运维命令，让你能查状态、冻结开仓、暂停和恢复。

status

查状态

任何时候不确定，就先查状态。期望看到 NORMAL。

freeze-open

冻结开仓

你觉得有风险但还没到全面暂停，就先禁止新的开仓。

halt

暂停系统

状态未知、风险未知、对账异常时，先停下来。

resume

恢复

恢复前必须做账户、持仓、活跃订单检查，并引用 OK 的对账编号。

手动冻结开仓 
cd /Users/shenxi/Desktop/WORK-SPACE/个人量化系统/quant
.venv/bin/python scripts/ops.py \
  --store runtime/paper/meta.db \
  --events runtime/paper/events.jsonl \
  --operator shenxi \
  freeze-open \
  --reason manual-caution
确认检查后恢复 
cd /Users/shenxi/Desktop/WORK-SPACE/个人量化系统/quant
.venv/bin/python scripts/ops.py \
  --store runtime/paper/meta.db \
  --events runtime/paper/events.jsonl \
  --operator shenxi \
  resume \
  --reason checked-and-safe \
  --precheck account \
  --precheck positions \
  --precheck active_orders \
  --reconciliation-seq 42

--reconciliation-seq 42 只是示例。真实恢复时，要从 events.jsonl 里找到最近一次状态为 OK 或 REPAIRED 的对账事件编号。

九、断连演练：故意制造一次“出事了”

断连演练是 M3b 必做项目。它验证系统能不能在网关断开时冻结开仓、发 CRIT 告警、重新对账，然后恢复 NORMAL。

运行带断连演练的 Paper 
cd /Users/shenxi/Desktop/WORK-SPACE/个人量化系统/quant
.venv/bin/python scripts/run_paper.py \
  --strategy config/strategies/dual_ma_510300_paper.yaml \
  --paper config/paper.yaml \
  --max-bars 20 \
  --disconnect-drill \
  --disconnect-reason user-flow-demo

你应该看到什么

命令输出里应该有 final state: NORMAL，并打印 disconnect drill。 事件日志里应该能看到 gateway_disconnect、alert、reconciliation、recovery。

十、M3b：真钱前 10 个交易日观察

M3b 是从“系统能跑”到“系统值得接近实盘”的过渡。它要求至少 10 个交易日，每天有记录、每天对账零差异，并且完成断连和 CRIT 告警演练。

M4 之前不要接真金白银

只有 M3b 签收后，才有资格考虑 M4 的 QMT / 实盘接入。否则你只是在没有足够刹车和记录的情况下冒险。

08:45 盘前

启动前检查

确认测试通过、配置是 paper、启动 Paper、查看状态是 NORMAL。

盘中

盯住异常

如果出现 CRIT 告警，先查事件日志和状态库。需要谨慎时用 freeze-open。

15:05 收盘

收盘记录

确认订单、成交、拒单、对账结果、告警，并归档 runtime 记录。

十一、常见错误：看到这些别慌

新手最怕报错。下面这些是最常见的坑，大部分不是系统坏了，而是目录、命令或状态没对。

你看到的问题 通常原因 怎么处理
command not found: python 系统没有 python 这个命令。 用 .venv/bin/python，或先进入虚拟环境。
提示找不到 scripts/run_backtest.py 你没有进入 quant 目录。 先运行 cd /Users/shenxi/Desktop/WORK-SPACE/个人量化系统/quant。
缺少 pandas、pydantic 等依赖 没有使用项目里的虚拟环境。 确认命令前缀是 .venv/bin/python。
status 不是 NORMAL 系统可能处于冻结或暂停状态。 先看 runtime/paper/events.jsonl，确认最近对账 OK 后再考虑 resume。
对账不是 OK 现金或持仓可能出现差异。 不要恢复交易，先记录差异，查清原因。
误删 runtime 运行记录被清掉。 普通练习可以重跑；M3b 连续观察期间则这天很可能不能计入。
回测没有成交 样例数据短，订单不一定撮合成交。 先确认流程产物完整；收益不是当前样例的重点。

十三、下一步边界：什么时候才考虑 M4

真正的纪律是：系统没有完成观察和签收，就不升级风险。M4 不是“技术上能写代码就开始”，而是 M3b 用事实证明系统可控之后才开始。

现在不要做

不要接真实券商和真钱

不要跳过 M3b，不要把 Paper 当实盘，不要在没有对账和告警闭环时接 QMT。

现在应该做

练流程、记日志、做观察

每天跑 Paper，记录状态、订单、成交、对账、告警，完成 10 日观察和演练。

最后一句话： 这个项目现在最有价值的地方，不是让你马上赚钱，而是让你建立一套可验证、可复盘、能刹车的量化交易工作流。 当你能解释每一条命令、每一个状态和每一次对账时，你才真的开始掌控系统。
