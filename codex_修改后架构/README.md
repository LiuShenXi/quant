## 这套架构的"宪法"(五条设计目标)

整个文档包的所有设计决策都服务于这五条,遇到取舍时按此优先级判断:

1. **回测实盘一致**:策略代码只写一份,在回测引擎和实盘引擎中行为一致。靠"策略契约"(03 号文档)保证。
2. **策略即插件**:策略只依赖契约,不依赖任何具体引擎、数据源、券商 SDK。你打磨好的策略放进 `strategies/` 目录、写一份 YAML 配置,即可接入。
3. **基础设施可替换**:数据源、回测引擎、券商通道都在抽象层后面,换任何一个都不动策略代码。
4. **单人可维护**:单机部署、模块化单体(不搞微服务)、文件型存储(Parquet + DuckDB + SQLite),一个人能看懂、能修。
5. **默认安全**:所有订单必须经过风控模块;进程崩溃重启后必须先对账再交易;任何异常默认"停下来报警"而不是"继续跑"。

## 全局约定

- 代码示例基于 Python 3.11+,带类型标注。
- 文中所有费率、门槛、涨跌幅等数字均为**示例值**,以你的券商和交易所实际规则为准,且全部做成配置而非硬编码。
- "策略层"(`strategies/` 目录及研究产出)是你自己的东西,本文档包不涉及;其余全部属于平台层,本包覆盖。
- 策略唯一允许依赖的平台包统一为 `core.contract`。示例代码、import-linter 规则和兼容性承诺全部以这个包名为准。
- 平台层 API 的兼容性承诺见 03 号文档第 8 节:对策略可见的接口只做增量演进,保证你后续打磨的策略能持续直接接入。

## v1.1 修订重点

1. **统一契约包名**:`StrategyBase`、`Context`、`Bar/Order/Trade/Position/Account` 全部放在 `core.contract`。
2. **收紧策略可见世界**:`mode` 不再暴露给策略逻辑,运行模式只进入配置快照、日志和报告。
3. **增强实盘订单模型**:订单和成交增加策略实例、账户、券商单号、拒单原因、更新时间等字段,支撑 OMS 对账和故障恢复。
4. **补硬数据语义**:明确 A股默认时区、bar 结束时间、复权因子口径、缺 bar/停牌/涨跌停字段。
5. **路线图更保守**:M0/M1 先交付最小可信闭环,OMS/QMT 等真钱相关能力后置到 paper 验证之后。
