# 04 · 事件驱动回测引擎

## 1. 定位:双轨制中的"严格轨"

- 向量化粗筛(vectorbt / notebook):快,用于因子探索和参数扫描,**结果不作为上线依据**。
- 本引擎(事件驱动):慢但语义与实盘一致(异步回报、部分成交、T+1、涨跌停),**只有这里的结果才能让一个策略获得上实盘的资格**。

引擎与实盘引擎共享:`core.contract`、DataService、风控模块、订单/持仓数据结构——差异仅在"订单交给撮合器"还是"交给券商网关"。

## 2. 主循环

```
加载配置 → 实例化策略 → on_init → on_start
按时间序合并所有标的的 bar 流(同一时刻按 symbol 排序,保证确定性)
for dt in 时间轴:
    1. 撮合器先处理上一轮挂单(用当前 bar 撮合)→ 产生 Trade
    2. 派发 on_order / on_trade 回报 → 更新持仓与现金
    3. 更新行情快照(ctx 的"当前时刻"推进到 dt)
    4. 派发 on_timer(若到时)
    5. 派发 on_bar → 策略产生新订单 → 风控检查 → 生成 Order(order_id/strategy_id/account_id) → 进入挂单队列
    6. 记录当日权益快照
on_stop → 生成绩效报告与流水
```

关键点:第 5 步产生的订单**最早在下一根 bar 才可能成交**——从机制上消灭"用收盘价信号、按收盘价成交"这种最常见的未来函数。回测生成的订单也必须填充与实盘一致的 `strategy_id/account_id/created_at/updated_at/reject_reason` 字段,这样 report 和黄金回归测试能覆盖同一份契约。

## 3. 撮合规则(A股默认配置,均可按市场重载)

| 规则 | 默认行为 |
|---|---|
| 成交时机 | t 时刻信号 → t+1 bar 撮合;市价单按 t+1 开盘价 ± 滑点 |
| 限价单 | bar 内价格触及即成交:买单 limit ≥ low、卖单 limit ≤ high;保守起见成交价取 min(limit, open)/max(limit, open) |
| 涨跌停 | 用 bars 表里的 limit_up/limit_down:开盘即涨停 → 买单不成交;跌停 → 卖单不成交(可配置"封板可部分成交"宽松模式) |
| 停牌 | suspended bar 不撮合,持仓按最后收盘价估值 |
| T+1 | 按 instrument.t_plus:当日买入计入 qty 但不计入 sellable,次日转可卖 |
| 整手 | 按 lot_size/qty_step 向下取整;不足最小单位的目标差额忽略 |
| 量能上限 | 单笔成交量 ≤ α × 当根 bar 成交量(默认 α=5%),超出部分挂到后续 bar(部分成交)——防止回测里"一口吃下流动性根本不够的仓位" |

撮合器是独立类 `Matcher`,所有规则有单元测试覆盖(见 08 测试策略)。

v1.1 补充:撮合器输入只接受 `core.contract.Order` 和 `core.contract.Bar`;撮合器输出 `Trade` 与状态更新事件。不要让策略或回测引擎直接改持仓,持仓只由成交事件驱动。

## 4. 成本模型(全部走配置,示例值仅为示意)

```yaml
# config/costs/cn_stock.yaml —— 数字以你的券商/最新税费为准
commission_rate: 0.00025      # 佣金,双边
commission_min: 5.0           # 最低佣金(部分券商免五,自行修改)
stamp_tax: 0.0005             # 印花税,仅卖出
transfer_fee: 0.00001         # 过户费,双边
slippage:
  model: fixed_bps            # fixed_bps | tick | none
  value: 5                    # 万分之五
```

要求:回测报告必须同时输出**双倍成本敏感性**(成本×2 再跑一遍的收益)——对成本敏感的策略,实盘大概率不及预期。

## 5. 账户核算

- 现金流:成交金额 ± 各项费用,挂单冻结对应资金/可卖数量。
- 分红送配:回测基于动态前复权价时,分红收益已隐含在价格序列中(等效全额再投资),适合研究级结论;若要求逐笔现金流精确(生产级账本),在除权日按因子调整持仓数量并入账现金分红——两种模式都实现,配置切换,报告中注明所用模式。
- 退市:delist_date 起强制按最后价清仓并记录。

## 6. 绩效与产出物

每次回测输出一个**自包含结果目录**:

```
results/{run_id}/
  config_snapshot.yaml     # 当次完整配置(含契约版本、params哈希)
  trades.csv               # 逐笔成交
  orders.csv               # 全部订单(含被拒原因)
  equity.csv               # 逐日权益、仓位、现金
  report.html              # 指标 + 净值曲线 + 回撤曲线
  events.jsonl             # 与实盘同结构的事件流水,用于重放/对比
```

指标至少包含:总/年化收益、最大回撤及持续期、夏普、Sortino、Calmar、胜率、盈亏比、换手率、平均持仓周期、暴露度,以及双倍成本敏感性结果。

## 7. 防自欺清单(每个策略上线前过一遍)

- [ ] 无未来函数:信号与成交隔 bar;`ctx.history` 物理截止到 now
- [ ] 无幸存者偏差:股票池含退市标的;指数成分用时点表(universe_membership)
- [ ] 复权正确:用动态前复权;基本面按公告日对齐
- [ ] 成交可实现:量能上限开启;涨跌停规则开启
- [ ] 成本真实:含税费滑点,且双倍成本下结论仍成立
- [ ] 参数稳健:最优参数邻域(±20%)绩效平滑,不是孤峰
- [ ] 样本外验证:留出最近 1~2 年做 walk-forward,样本外不参与任何调参

## 8. 黄金回归测试(守护引擎本身)

固定一份小数据集 + 一个固定策略 + 固定参数,把订单、成交、权益和事件流水存为基准文件;CI/每次改动核心层后重跑,**逐笔比对必须完全一致**。引擎任何无意的行为漂移(撮合顺序、取整方式、费率计算、字段填充)都会在这里被抓住——这是"平台层可以放心重构"的安全网。
