# 06 · 风控模块

## 1. 原则

1. **独立且不可绕过**:风控位于策略与 OMS 之间的唯一通道上,策略没有任何接口能跳过它。
2. **白名单思维**:没明确允许的就是禁止(标的白名单、模式白名单)。
3. **失败安全**:风控模块自身异常 → 拒单并告警,而不是放行。
4. 策略实例的 `risk:` 配置只能在全局限制内**收紧**,不能放宽。

## 2. 前置检查流水线(每笔订单,按序执行,任一失败即 REJECTED 并记录原因)

| # | 检查 | 默认规则(示例) |
|---|---|---|
| 1 | 标的白名单 | symbol 必须在该策略实例 universe 内 |
| 2 | 交易时段 | 当前时间在该交易所连续竞价时段 |
| 3 | 价格围栏(防胖手指) | 限价偏离最新价 > 2% → 拒 |
| 4 | 单笔上限 | 单笔金额 ≤ max_order_value |
| 5 | 持仓上限 | 成交后该标的市值 ≤ max_position_value;总仓位 ≤ max_gross_exposure |
| 6 | 资金/可卖检查 | 买:可用资金足够(含费用);卖:sellable 足够(T+1) |
| 7 | 频率限制 | 每分钟订单数 ≤ N;当日撤单比例 ≤ 阈值(同时满足程序化合规要求,防策略死循环刷单) |
| 8 | 自成交防护 | 同一标的同时存在买卖挂单且价格交叉 → 拒 |
| 9 | 全局状态 | Kill Switch 非 NORMAL 时按状态规则放行/拒绝 |

所有拒单必须返回结构化 `reject_reason` 和 `risk_rule_id`,由 OMS 写入 `Order.reject_reason`、事件流水和回测/实盘报告。不要只写日志,否则策略回调和事后复盘拿不到一致原因。

## 3. 账户级 Kill Switch(状态机)

```
NORMAL ──日内亏损≥L1──▶ FREEZE_OPEN(禁新开仓,允许降风险订单/撤单)
NORMAL/FREEZE ──日内亏损≥L2 或 对账失败──▶ HALT(全部拒单,撤掉所有挂单)
HALT ──人工确认后──▶ NORMAL      # 自动恢复被禁止:必须人看过才能继续
可选:FLATTEN(一键清仓)由人工命令触发,不自动触发
```

L1/L2 示例:日内回撤 2% / 4%(配置项)。状态变更立即推送最高级别告警。

## 4. 运行时监控(独立于订单路径的看门狗线程)

| 监控 | 触发 | 动作 |
|---|---|---|
| 行情新鲜度 | 交易时段内 N 秒无任何 tick/bar | 告警;持续超时 → FREEZE_OPEN |
| 网关心跳 | 连接断开 | 告警 + 冻结开仓,走 05 的重连流程 |
| 持仓漂移 | 盘中对账差异 | 见 05 对账规则 |
| 策略异常 | 回调抛未捕获异常 | 隔离该策略实例(停止派发事件),其余策略不受影响,告警 |

## 5. 风控配置示例

```yaml
# config/risk/global.yaml
whitelist_mode: true
price_collar_pct: 0.02
max_order_value: 200_000
max_position_value_per_symbol: 500_000
max_gross_exposure_pct: 0.95        # 总持仓市值/总资产
max_orders_per_minute: 10
max_cancel_ratio_daily: 0.5
kill_switch:
  daily_loss_freeze_pct: 0.02       # L1
  daily_loss_halt_pct: 0.04        # L2
market_data_staleness_sec: 60
```

## 6. 人工干预

提供一个最小命令行工具(`scripts/ops.py`):查看状态、手动 FREEZE/HALT/恢复、一键撤全部挂单、一键清仓(二次确认)。所有人工操作同样写入事件流水。手机端通过告警消息知道"出事了",处理动作回到电脑上做——不要做"回复消息就能下单"之类的快捷通道,误触风险大于便利。

人工恢复必须写入操作者、时间、原因和恢复前检查项。即使是个人系统,也不要让 Kill Switch 自动从 HALT 恢复。
